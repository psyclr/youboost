# YouBoost — Гайд для админа

## Общая картина

Ты — оператор маркетплейса. Твоя задача:

1. **Подключить поставщиков** — внешние SMM-панели, которые выполняют заказы
2. **Настроить каталог** — какие услуги продавать и по какой цене
3. **Следить за системой** — заказы, пользователи, баланс поставщиков
4. **Решать проблемы** — рефанды, принудительные смены статусов, блокировки

Доступ в админку — только у пользователей с ролью `ADMIN`.

---

## Дашборд (`/admin`)

Обзорная страница. Четыре карточки:

| Метрика         | Что показывает                                          |
| --------------- | ------------------------------------------------------- |
| Total Users     | Сколько всего зарегистрировано                          |

Плюс таблица последних 10 заказов — быстро видно, что происходит.

## Поставщики — как подключить

### Что такое поставщик?

Поставщик — это внешний SMM-сервис с API. У него есть ресурсы для накрутки
просмотров/лайков/подписчиков. YouBoost отправляет ему заказы и опрашивает статус.

Почти все панели используют **одинаковый формат API** (индустриальный стандарт),
поэтому подключение любого поставщика выглядит одинаково.

### Как подключить

Отправь запрос `POST /providers`:

```json
{
  "name": "SMM Panel Alpha",
  "apiEndpoint": "https://smmpanel.example.com/api/v2",
  "apiKey": "abc123secretkey",
  "priority": 10
}
```

| Поле        | Что это                                                   |
| ----------- | --------------------------------------------------------- |
| name        | Название для себя (видно только в админке)                |
| apiEndpoint | URL API поставщика (он даёт при регистрации)              |
| apiKey      | Ключ API (поставщик выдаёт в личном кабинете)             |
| priority    | Приоритет выбора: чем выше число, тем первее используется |
| metadata    | Любые заметки в JSON (необязательно)                      |

API-ключ **шифруется AES-256** перед сохранением в базу. Никогда не хранится открытым текстом.

### Несколько поставщиков

Можно подключить сколько угодно. Система выбирает **самого приоритетного активного**:

```
Provider A (priority: 10, active) ← выбирается первым
Provider B (priority: 5, active)  ← запасной
Provider C (priority: 3, inactive) ← не используется
```

Если Provider A упадёт (circuit breaker сработает) — заказы продолжат уходить на Provider B.

### Circuit Breaker

Автоматическая защита от зависших поставщиков:

- Если поставщик **5 раз подряд** не ответил или вернул ошибку —
  система **перестаёт его дёргать на 60 секунд**
- Через 60 секунд пробует снова
- Один успешный запрос сбрасывает счётчик ошибок

Это предотвращает ситуацию "поставщик лежит, а мы долбим его 1000 раз в минуту".

### Stub-режим (для разработки)

Если в `.env` стоит `PROVIDER_MODE=stub`:

- Заказы **не уходят** на реальных поставщиков
- Используется мок: принимает любой заказ, но статус навсегда остаётся PROCESSING
- Полезно для тестирования интерфейса, биллинга, вебхуков без реальных трат

Для боевой работы: `PROVIDER_MODE=real` + хотя бы один активный поставщик.

### Как поставщик выполняет заказ (что происходит внутри)

```
1. Пользователь нажал "Заказать 5000 просмотров"
2. YouBoost выбирает поставщика (по приоритету)
3. Отправляет HTTP-запрос на API поставщика:

   POST https://smmpanel.example.com/api/v2
   key=abc123&action=add&service=123&link=youtube.com/...&quantity=5000

4. Поставщик отвечает: { "order": 78901 }
5. Каждые 30 секунд YouBoost спрашивает:

   POST https://smmpanel.example.com/api/v2
   key=abc123&action=status&order=78901

6. Поставщик отвечает: { "status": "completed", "remains": 0 }
7. YouBoost обновляет статус → списывает деньги → шлёт уведомление
```

## Каталог — настройка услуг

### Создание услуги

На странице `/admin/services` нажми "Add Service":

| Поле           | Пример                  | Объяснение                              |
| -------------- | ----------------------- | --------------------------------------- |
| Name           | YouTube Views — Premium | Что увидит клиент                       |
| Platform       | YOUTUBE                 | Площадка                                |
| Type           | VIEWS                   | Тип услуги                              |
| Price per 1000 | 1.50                    | **Твоя цена** (наценка над поставщиком) |
| Min Quantity   | 100                     | Минимальный заказ                       |
| Max Quantity   | 100000                  | Максимальный заказ                      |
| Description    | Fast delivery, 24h      | Описание (необязательно)                |

### Ценообразование (твоя маржа)

Главное — **Price per 1000 в каталоге =/= цена поставщика**.

```
Поставщик продаёт тебе: $0.50 за 1000 просмотров
Ты ставишь в каталог:   $1.50 за 1000 просмотров
Твоя маржа:             $1.00 за каждые 1000
```

Клиент видит только твою цену. Цена поставщика — это твоя себестоимость.

### Активация / Деактивация

- **Деактивировать**: услуга пропадает из каталога, новые заказы невозможны
- Существующие заказы продолжат выполняться
- **Активировать обратно**: услуга снова появляется в каталоге

### Редактирование

Можно менять любые поля: цену, лимиты, название. Изменения применяются сразу.
Уже созданные заказы сохраняют цену на момент создания.

## Управление пользователями

### Список пользователей (`/admin/users`)

Таблица с колонками: Username, Email, Role, Status, Дата регистрации.

Фильтры:

- **По роли**: USER (обычный), RESELLER (реселлер), ADMIN
- **По статусу**: ACTIVE, SUSPENDED, BANNED

### Детали пользователя (`/admin/users/[id]`)

#### Смена роли

| Роль     | Что даёт                                                           |
| -------- | ------------------------------------------------------------------ |
| USER     | Обычный пользователь — может покупать услуги                       |
| RESELLER | Реселлер — то же что USER (пока без доп. функций, роль на будущее) |
| ADMIN    | Полный доступ к админке                                            |

Выбрал в выпадающем списке — сохранилось мгновенно.

#### Смена статуса

| Статус    | Эффект                                      |
| --------- | ------------------------------------------- |
| ACTIVE    | Всё работает нормально                      |
| SUSPENDED | Временная блокировка (можно разблокировать) |
| BANNED    | Полная блокировка                           |

#### Корректировка баланса

Два поля: **Сумма** и **Причина**.

```
Сумма: +50    Причина: "Компенсация за сбой при заказе #abc123"
Сумма: -10    Причина: "Штраф за нарушение правил"
Сумма: +100   Причина: "Промо-бонус для нового клиента"
```

- Положительная сумма = начисление
- Отрицательная сумма = списание
- Причина **обязательна** — записывается в ledger для аудита
- Тип транзакции: ADMIN_ADJUSTMENT

## Управление заказами

### Список заказов (`/admin/orders`)

Видны **все** заказы в системе (не только свои). Фильтр по статусу.

Колонки: Order ID, User ID, Status, Quantity, Price, Date, Actions.

### Принудительная смена статуса

Кнопка "Status" на строке заказа. Выбираешь любой статус из списка.

**Когда это нужно:**

- Поставщик выполнил заказ, но API не обновил статус (зависло на PROCESSING)
  → Принудительно ставишь COMPLETED
- Поставщик отменил заказ на своей стороне, но у нас висит PROCESSING
  → Принудительно ставишь FAILED или CANCELLED

**Что происходит при смене:**

- Для терминальных статусов (COMPLETED, PARTIAL, FAILED, CANCELLED, REFUNDED)
  автоматически проставляется `completedAt`
- Для нетерминальных (PENDING, PROCESSING) — `completedAt` обнуляется

**Важно:** принудительная смена статуса **НЕ трогает деньги автоматически**.
Если заказ завис на PROCESSING и ты ставишь COMPLETED, деньги останутся в hold.
Для возврата денег используй рефанд.

### Рефанд

Кнопка "Refund" на строке заказа. Подтверждаешь — и:

1. Деньги возвращаются на баланс пользователя (из hold в available)
2. Статус заказа меняется на REFUNDED
3. В ledger создаётся запись типа REFUND
4. Повторный рефанд невозможен (защита от дублей)

**Когда делать рефанд:**

- Заказ завис и поставщик не может выполнить
- Клиент пожаловался на качество
- Произошёл сбой в системе

## Мониторинг системы

### На что обращать внимание

**1. Заказы в PROCESSING слишком долго**

Если заказ висит в PROCESSING больше суток — скорее всего проблема с поставщиком:

- Поставщик лежит (проверь его сайт/API)
- Circuit breaker сработал (подожди 60 секунд или перезапусти)
- Stub-режим включён (`PROVIDER_MODE=stub` — заказы никогда не завершатся)

**2. Много FAILED заказов подряд**

Возможные причины:

- Поставщик закончился (нет ресурсов для выполнения)
- API-ключ просрочился или отозван
- Поставщик заблокировал аккаунт
- Некорректный service ID

**3. Баланс поставщика**

В модели Provider есть поле `balance` — но оно **не обновляется автоматически**.
Проверяй баланс на панели поставщика вручную. Если денег нет —
заказы будут приниматься, но не выполняться.

### Логи

Система пишет структурированные логи через Pino (JSON-формат):

- Каждый HTTP-запрос логируется (метод, URL, ID запроса)
- Ошибки логируются с полным контекстом
- Фоновые воркеры (polling, webhook delivery) тоже пишут логи

## Типичные сценарии

### Сценарий: Подключение нового поставщика

```
1. Регистрируешься на SMM-панели (например, justanotherpanel.com)
2. В личном кабинете находишь API Key и API URL
3. POST /providers:
   {
     "name": "JustAnotherPanel",
     "apiEndpoint": "https://justanotherpanel.com/api/v2",
     "apiKey": "ключ_отсюда",
     "priority": 10
   }
4. В .env ставишь PROVIDER_MODE=real
5. Перезапускаешь сервер
6. Готово — новые заказы пойдут на этого поставщика
```

### Сценарий: Клиент жалуется что заказ не выполнен

```
1. Находишь заказ в /admin/orders
2. Смотришь статус:
   - PROCESSING → проверяешь поставщика, ждёшь, или делаешь рефанд
   - FAILED → деньги уже вернулись автоматически
   - COMPLETED → проверяешь реальный результат на YouTube/Instagram
3. Если нужен возврат → кнопка "Refund"
```

### Сценарий: Нужно временно убрать услугу

```
1. /admin/services → находишь услугу
2. Нажимаешь Toggle → "Deactivate"
3. Услуга пропадает из каталога
4. Активные заказы по ней продолжат выполняться
5. Когда готов — активируешь обратно
```

### Сценарий: Нужно дать бонус клиенту

```
1. /admin/users → находишь пользователя → переходишь в детали
2. В блоке "Wallet" вводишь:
   - Amount: 25
   - Reason: "Welcome bonus for new customer"
3. Нажимаешь "Adjust"
4. На балансе клиента появляется +$25
```

### Сценарий: Нужно заблокировать мошенника

```
1. /admin/users → находишь пользователя
2. Меняешь Status → BANNED
3. Пользователь больше не может войти и создавать заказы
4. Его активные заказы продолжат выполняться
   (если нужно остановить — принудительно меняй статусы на CANCELLED + рефанд)
```

## Переменные окружения (важные для админа)

| Переменная                  | Что значит                                               | Пример                           |
| --------------------------- | -------------------------------------------------------- | -------------------------------- |
| PROVIDER_MODE               | stub (тест) или real (боевой)                            | `real`                           |
| PROVIDER_ENCRYPTION_KEY     | Ключ для шифрования API-ключей поставщиков (32+ символа) | `my-super-secret-key-32-chars!!` |
| ORDER_POLL_INTERVAL_MS      | Как часто опрашивать поставщиков (мс)                    | `30000` (30 сек)                 |
| CIRCUIT_BREAKER_THRESHOLD   | Сколько ошибок до блокировки поставщика                  | `5`                              |
| CIRCUIT_BREAKER_COOLDOWN_MS | Сколько ждать после блокировки (мс)                      | `60000` (60 сек)                 |
| RATE_LIMIT_MAX              | Глобальный лимит запросов (на всех пользователей)        | `100`                            |
| RATE_LIMIT_WINDOW_MS        | Окно лимита (мс)                                         | `60000` (1 мин)                  |

## API-эндпоинты админки (справочник)

### Дашборд

```
GET /admin/dashboard/stats
```

### Пользователи

```
GET    /admin/users                       Список (фильтры: role, status, page, limit)
GET    /admin/users/:userId               Детали + кошелёк
PATCH  /admin/users/:userId               Изменить роль/статус
POST   /admin/users/:userId/balance/adjust Корректировка баланса
```

### Заказы

```
GET    /admin/orders                      Список (фильтры: status, userId, page, limit)
GET    /admin/orders/:orderId             Детали
PATCH  /admin/orders/:orderId/status      Принудительная смена статуса
POST   /admin/orders/:orderId/refund      Рефанд
```

### Каталог услуг

```
GET    /admin/services                    Список всех услуг
POST   /admin/services                    Создать услугу
PATCH  /admin/services/:serviceId         Обновить услугу
DELETE /admin/services/:serviceId         Деактивировать
```
